\begin{abstract}
Dynamic invariant detectors such as Daikon infer likely program invariants
by evaluating candidate properties against execution traces.
However, Daikon's candidates are drawn from a fixed template library,
which prevents it from expressing properties that involve project-specific
logic or more than three variables.
We present \textsc{Oca}, a dynamic invariant detector that uses a large language
model (LLM) to propose candidates instead of deriving them from templates.
LLM-proposed invariants are more expressive, but their integration into
a dynamic analysis pipeline raises non-trivial research challenges
involving context selection, semantic safety, and scalability.
We address these challenges and evaluate \textsc{Oca} on the Defects4J benchmark,
showing that it infers invariants beyond Daikon's expressive reach
and that these invariants improve bug-detection capability in property-based testing.
\end{abstract}

\section{Problem and Motivation}

Program invariants---properties that hold at every execution of a program
point---are foundational to testing, verification, and program understanding~\cite{ernst2007daikon}.
Dynamic invariant detectors such as Daikon~\cite{ernst2007daikon}
infer likely invariants automatically from execution traces,
avoiding the burden of manual specification.
Daikon is widely used in research and practice~\cite{endres2024llm,sun2025classinvgen,pei2023llm,cao2025clause2inv,ma2025specgen},\todo{Not all of these use Daikon as a baseline, and in fact this is the same list that you use below of LLM-based invariant generators. Did you mean to copy-paste a different list, maybe? Certainly clause2inv, today's RG paper, does not mention Daikon.}
but its design imposes a hard expressive ceiling:
it generates candidates from a fixed template library,
so any invariant that cannot be expressed within that grammar
is never even considered.
Concretely, Daikon cannot express (1)~properties involving
project-specific methods or domain abstractions, or
(2)~relations among more than three variables.
These limitations matter in practice:
real programs frequently contain semantic constraints
that cut across these boundaries.
Prior work has extended Daikon with new template families~\cite{le2019sling},
but adding templates does not solve the expressiveness problem in general:
Daikon's run-time cost scales with the number of candidates,
so template libraries must remain small and fixed.

Recent work has begun to explore LLMs as a source of invariant candidates~\cite{endres2024llm,pei2023llm,cao2025clause2inv,ma2025specgen,sun2025classinvgen},
but prior efforts were manual, limited in scope, or restricted to specific invariant classes
and did not produce a validated, end-to-end tool for method-level pre- and post-conditions.
We fill this gap with \textsc{Oca}, a dynamic invariant detector
that replaces Daikon's template library with LLM-generated candidates
while preserving Daikon's execution-based validation model.

\section{Research Challenges}

Substituting an LLM for a template library is not straightforward.
We identify five challenges that any such system must address.
The challenges fall into two groups.
The first concerns eliciting useful output from the LLM:
without careful choices of context and prompt design,
models produce vacuous or irrelevant candidates (C1, C2).
The second, and more fundamental, concerns the fact that LLM-generated invariants
can be poorly behaved in ways that manually-chosen templates never are---they
may be syntactically invalid, corrupt the validation signal through side effects,
or fail to terminate (C3, C4, C5).

\paragraph{C1: Providing effective context.}
LLMs are sensitive to the context supplied in their prompts.
Too little context yields trivial or irrelevant invariants;
too much wastes token budget and can distract the model.
The right granularity is not obvious and cannot be determined a priori.
We conducted an open-coding analysis of 187 recent LLM-based SE papers from top venues
to derive a taxonomy of context strategies used in practice,
then evaluated each strategy on three real-world Java projects---
Apache Hudi~\cite{hudi}, libGDX~\cite{libgdx}, and RxJava~\cite{rxjava}---
to select the configuration that maximizes held invariants.

\paragraph{C2: Designing prompts that elicit well-formed invariants.}
Without careful prompt design, an LLM may produce vacuous,
trivially true, or ill-structured properties.
We derived six candidate strategies from established taxonomies of LLM prompting
for code tasks---baseline direct prompting, few-shot examples, chain-of-thought,
stepwise decomposition, self-refinement, and multi-sample agreement---
and evaluated each on the same three projects~\cite{hudi,libgdx,rxjava},
fixing the context strategy from C1 throughout.
Few-shot prompting produced the most held invariants across all three projects
and was selected as the default configuration.

\paragraph{C3: Handling syntactically invalid candidates.}
\textsc{Oca} filters candidates at three stages.
Before injection, a deterministic filter rejects expressions that are trivially true,
contain bare assignments, reference unknown identifiers,
use constructs requiring additional imports, apply null comparisons to primitives,
or fail JavaParser's expression parser.
After injection, the instrumented project is compiled iteratively:
begin/end markers localize each failing guard, which is commented out and retried;
if an error cannot be resolved by removal, the file is restored from the original source.
After execution, candidates that cause test failures are grouped by method,
batched, and iteratively disabled until the suite passes,
identifying the minimal set responsible for failures.

\paragraph{C4: Preventing invariant checks from corrupting the validation signal.}
Each injected guard checks a per-thread re-entrancy flag before evaluating its expression.
If the flag is already set---meaning another check is active on the same thread---the
guard skips evaluation entirely.
Without this, evaluating one invariant could call methods with their own injected guards,
which would execute against artificial inputs and be falsified incorrectly.

\paragraph{C5: Handling non-terminating candidates.}
\textsc{Oca} detects non-termination per invariant using two mechanisms in a background monitor.
The first tracks the UUID of whichever invariant is currently mid-evaluation
via a shared status field and kills the run if that field is unchanged beyond a configurable threshold.
The second catches deadlocks between invariant calls---where the status field would already
be clear---by watching for total log silence over the same threshold.
In both cases, the offending UUID is appended to a persistent exclusion file;
on the next run, every guard loads this file at JVM startup and skips listed invariants
across all injection points without source modification.
A hard wall-clock limit serves as a final safety net.

\section{Approach: \textsc{Oca}}

\textsc{Oca} follows the same execution-based validation paradigm as Daikon:
candidates are proposed, checked against concrete executions,
and only those consistent with observed behavior are reported.
The key difference is the proposal module.

For each method entry and exit point,
\textsc{Oca} queries an LLM with a structured prompt
containing the selected source context.
Generated candidates are filtered and deduplicated,
then inserted into a working copy of the source code as guarded run-time checks.
Each check is wrapped in a try-catch block with a depth guard
to address C4 and C5;
the program is then compiled iteratively to address C3.
Finally, the instrumented program is executed on its existing test suite.
Invariants that compile, execute at least once, and are never falsified are reported as held.

A key structural difference from Daikon is that
\textsc{Oca} generates far fewer candidates per program point
than Daikon's $2^{64}$-scale template enumeration,
so it does not require the statistical machinery
(comparability analysis, implication pruning)
that Daikon uses to reduce its output.
This simplifies the validation model
while the LLM's expressiveness compensates for the reduced candidate volume.

Across projects, an average of 16\% of held invariants involve project-specific logic

Ongoing evaluation on buggy Defects4J versions measures
whether these expressive invariants improve bug-detection capability
in property-based testing relative to Daikon's invariants.

\section{Contributions}

This work makes the following contributions:

\begin{itemize}
  \item A characterization of the research challenges in integrating LLMs
        into a dynamic invariant detection pipeline (C1--C5).
  \item \textsc{Oca}, an end-to-end tool that addresses these challenges
        for Java programs, supporting both cloud-hosted and locally hosted LLMs.
  \item An empirical evaluation on Defects4J showing that \textsc{Oca}
        infers invariants beyond Daikon's expressive reach
        and that those invariants improve downstream bug detection.
\end{itemize}
