\begin{abstract}
Dynamic invariant detectors such as Daikon infer likely program invariants
by evaluating candidate properties against execution traces.
However, Daikon's candidates are drawn from a fixed template library,
which prevents it from expressing properties that involve project-specific
logic or more than three variables.
We present \textsc{Oca}, a dynamic invariant detector that uses a large language
model (LLM) to propose candidates instead of deriving them from templates.
LLM-proposed invariants are more expressive, but their integration into
a dynamic analysis pipeline raises non-trivial research challenges
involving context selection, semantic safety, and scalability.
We address these challenges and evaluate \textsc{Oca} on the Defects4J benchmark,
showing that it infers invariants beyond Daikon's expressive reach
and that these invariants improve bug-detection capability in property-based testing.
\end{abstract}

\section{Problem and Motivation}

Program invariants---properties that hold at every execution of a program
point---are foundational to testing, verification, and program understanding.
Dynamic invariant detectors such as Daikon~\cite{ernst2007daikon}
infer likely invariants automatically from execution traces,
avoiding the burden of manual specification.
Daikon is widely used in research and practice,
but its design imposes a hard expressive ceiling:
it generates candidates from a fixed template library,
so any invariant that cannot be expressed within that grammar
is never even considered.
Concretely, Daikon cannot express (1)~properties involving
project-specific methods or domain abstractions, or
(2)~relations among more than three variables.
These limitations matter in practice:
real programs frequently contain semantic constraints
that cut across these boundaries.

Recent work has begun to explore LLMs as a source of invariant candidates~\cite{endres2024llm},
but prior efforts were manual or limited in scope
and did not produce a validated, end-to-end tool.
We fill this gap with \textsc{Oca}, a dynamic invariant detector
that replaces Daikon's template library with LLM-generated candidates
while preserving Daikon's execution-based validation model.

\section{Research Challenges}

Substituting an LLM for a template library is not straightforward.
We identify four challenges that any such system must address.

\paragraph{C1: Providing effective context.}
LLMs are sensitive to the context supplied in their prompts.
Too little context yields trivial or irrelevant invariants;
too much wastes token budget and can confuse the model.
The right granularity is not obvious and must be determined empirically.
\textsc{Oca} evaluates context configurations ranging from method signatures
to call-graph-extended class context,
using an open-coding analysis of 187 recent LLM-based SE papers
to identify the candidate strategies.

\paragraph{C2: Designing prompts that elicit well-formed invariants.}
Unlike templates, LLM output is unconstrained in form.
Without careful prompt engineering,
the model may produce vacuous, trivially true, or ill-structured properties.
\textsc{Oca} evaluates six prompting strategies derived from established
taxonomies of LLM prompting for code tasks,
selecting the best-performing configuration empirically
before proceeding to the main evaluation.

\paragraph{C3: Handling syntactically invalid candidates.}
LLM-generated expressions may reference out-of-scope identifiers,
violate typing constraints, or fail to compile when inserted into the program.
\textsc{Oca} addresses this through multi-stage filtering:
candidates are checked for syntactic validity and scope before instrumentation,
and compilation is performed iteratively,
disabling only the invariants implicated in each error until compilation succeeds.

\paragraph{C4: Ensuring semantic safety at runtime.}
Even syntactically valid invariants can be semantically unsafe:
they may invoke methods with side effects, throw exceptions,
or trigger nontermination through recursion.
\textsc{Oca} guards against these problems at runtime
by limiting re-entrant evaluation via a per-thread depth counter,
imposing a global time limit on test execution,
and treating any invariant that causes a test failure as falsified.

\section{Approach: \textsc{Oca}}

\textsc{Oca} follows the same execution-based validation paradigm as Daikon:
candidates are proposed, checked against concrete executions,
and only those consistent with observed behavior are reported.
The key difference is the proposal module.

For each method entry and exit point,
\textsc{Oca} queries an LLM with a structured prompt
containing the selected source context.
Generated candidates are filtered and deduplicated,
then inserted into a working copy of the source code as guarded runtime checks.
Each check is wrapped in a try-catch block with a depth guard
to address C4;
the program is then compiled iteratively to address C3.
Finally, the instrumented program is executed on its existing test suite.
Invariants that compile, execute at least once, and are never falsified are reported as held.

A key structural difference from Daikon is that
\textsc{Oca} generates far fewer candidates per program point
than Daikon's $2^{64}$-scale template enumeration,
so it does not require the statistical machinery
(comparability analysis, implication pruning)
that Daikon uses to reduce its output.
This simplifies the validation model
while the LLM's expressiveness compensates for the reduced candidate volume.

Across projects, an average of 16\% of held invariants involve project-specific logic
and 1.78\% relate more than three variables---both categories
entirely outside Daikon's expressive reach.
These are conservative figures: they exclude other forms of semantic richness
(e.g., nested logical structure) that also exceed Daikon's grammar.

Ongoing evaluation on buggy Defects4J versions measures
whether these expressive invariants improve bug-detection capability
in property-based testing relative to Daikon's invariants.

\section{Contributions}

This work makes the following contributions:

\begin{itemize}
  \item A characterization of the research challenges in integrating LLMs
        into a dynamic invariant detection pipeline (C1--C4).
  \item \textsc{Oca}, an end-to-end tool that addresses these challenges
        for Java programs, supporting both cloud-hosted and locally hosted LLMs.
  \item An empirical evaluation on Defects4J showing that \textsc{Oca}
        infers invariants beyond Daikon's expressive reach
        and that those invariants improve downstream bug detection.
\end{itemize}
